# Design: Persona Status via Hive Submodule + `go generate`

**Date:** 2026-04-17
**Status:** Approved; implementation plan to follow
**Supersedes:** [transpara-ai/site#14](https://github.com/transpara-ai/site/pull/14) (`feat/persona-status-sync`)
**Implements:** the "submodule + `go generate`" stopgap referenced in [hive's debt doc](https://github.com/transpara-ai/hive/blob/main/docs/debt/role-persona-decomposition.md)

## Problem

Site's `graph/personas/*.md` files do not carry the `<!-- Status: X -->` metadata that hive's `agents/*.md` files use to declare role maturity (`designed`, `running`, `absorbed`, `retired`, etc.). Consequently, `SeedAgentPersonas` cannot downgrade `Active=false` for absorbed or retired personas — the Agents UI can show roles the hive has already phased out.

The previous attempt (PR #14) solved this by *copying* each hive agent's metadata comment block into the corresponding site `.md` file — 54 files modified, one per persona. That is the drift risk the debt doc names explicitly: two copies of the same fact, hand-maintained.

## Goals

1. Hive's `agents/*.md` is the single source of truth for persona Status.
2. Site consumes Status at build time, not runtime, so `go build` has no new network or filesystem dependency.
3. `Active=false` is forced for personas whose hive-side Status is `absorbed` or `retired`.
4. CI fails if the generated output is out of date with the pinned hive commit.

## Non-goals

- Decomposing Role/Persona into separate files (the long-term fix the debt doc tracks — this is the stopgap).
- Runtime enforcement of status (blocking agent spawns when `status=retired`). Deferred.
- Importing `Absorbs`, `Absorbed-By`, `Reason`, or `Council-*` metadata into site. Scope is Status only.
- Auto-bumping the hive submodule pin. Manual, human-reviewed.
- Backwards compatibility with PR #14's 54-file propagation. Those edits are reverted.

## Components

### 1. Hive as a git submodule

- **Path:** `third_party/hive`
- **Remote:** `https://github.com/transpara-ai/hive.git`
- **Pinning:** manual. Bump with:
  ```
  cd third_party/hive && git fetch && git checkout origin/main && cd -
  go generate ./graph/personas/...
  git add third_party/hive graph/personas/status_gen.go
  git commit
  ```
- **Scope:** full submodule. Hive is small; sparse-checkout adds complexity for no benefit.
- **Ignore:** add `third_party/` to `.dockerignore` so build images don't pick up the checkout (the generated `.go` file is all the binary needs).

### 2. Generator tool (`cmd/gen-persona-status/main.go`)

A single-purpose Go program, ~80 lines, no external deps beyond the stdlib.

- **Flags (all paths resolved relative to CWD, which `go generate` sets to the directive's directory — `graph/personas/`):**
  - `-src` (default `../../third_party/hive/agents`) — directory of hive `agents/*.md`.
  - `-out` (default `status_gen.go`) — output file path.
- **Behavior:**
  1. List `*.md` entries in `-src`.
  2. For each file: read the leading ~20 lines, regex-match `^<!--\s*Status:\s*(\w+)\s*-->\s*$`. Persona name = filename without `.md`.
  3. **If no Status comment, skip the file** (don't error). Hive's `CONTEXT.md` and any future non-persona markdown are tolerated automatically. Print skipped filenames to stderr for visibility.
  4. Sort by persona name for deterministic output.
  5. Write Go file via `text/template`:
     ```go
     // Code generated by gen-persona-status; DO NOT EDIT.
     package personas

     // HiveStatus maps persona name → hive-declared lifecycle status.
     // Source: third_party/hive/agents/<name>.md <!-- Status: X -->
     var HiveStatus = map[string]string{
         "advocate":  "running",
         "allocator": "designed",
         // ...sorted...
     }
     ```
- **No runtime dependency on the submodule.** Generator only runs under `go generate`, not `go build`. The committed `status_gen.go` is the only artifact `go build` sees.

### 3. `//go:generate` directive (`graph/personas/generate.go`)

```go
package personas

//go:generate go run ../../cmd/gen-persona-status
```

One file, three lines. Invoked by `go generate ./graph/personas/...` or `go generate ./...`.

### 4. Consumer changes (`graph/personas.go`)

- `parsePersonaFile` keeps its current `(display, description)` signature. **No status parsing from `.md`.**
- `SeedAgentPersonas`:
  ```go
  status, ok := HiveStatus[name]
  if !ok {
      status = "ready"
  }
  active := personaActive[name]
  if status == "absorbed" || status == "retired" {
      active = false
  }
  // ...existing UPSERT, now threading status...
  ```

### 5. Store changes (`graph/store.go`) — re-landed verbatim from PR #14

- Add `Status string` field to `AgentPersona`.
- Boot migration: `ALTER TABLE agent_personas ADD COLUMN IF NOT EXISTS status TEXT NOT NULL DEFAULT 'ready';`
- Thread `status` through all three persona `SELECT` statements and the `UPSERT` in `SeedAgentPersonas`.

### 6. CI wiring (`.github/workflows/ci.yml`)

Three changes to the existing workflow:

1. `actions/checkout@v4` gains:
   ```yaml
   with:
     submodules: recursive
   ```
   so `third_party/hive` is populated on runners.

2. New step, inserted after "Install templ", before "Generate templates":
   ```yaml
   - name: Generate persona status
     run: go generate ./graph/personas/...
   ```

3. Existing "Check for uncommitted generated file changes" step broadens its glob:
   ```yaml
   run: git diff --exit-code -- '*_templ.go' '*_gen.go'
   ```

## Data flow

```
hive/agents/treasurer.md  <-- source of truth: <!-- Status: designed -->
     │
     │  (git submodule pin)
     ▼
third_party/hive/agents/treasurer.md
     │
     │  (go generate → gen-persona-status)
     ▼
graph/personas/status_gen.go:  HiveStatus["treasurer"] = "designed"
     │
     │  (go build, embedded in binary)
     ▼
SeedAgentPersonas(ctx):
     status := HiveStatus["treasurer"]   // "designed"
     active := personaActive["treasurer"] // existing hardcoded map
     if status ∈ {absorbed, retired} { active = false }
     UPSERT agent_personas (name, status, active, ...)
     │
     ▼
agent_personas table:  treasurer | designed | active
     │
     ▼
/agents UI
```

## Error handling

| Condition | Behavior |
|---|---|
| Generator: hive `.md` has no `Status` comment | Skip silently (with stderr log). Tolerates non-persona docs like `CONTEXT.md`. |
| Generator: `third_party/hive/agents` missing | Fail with message: "run `git submodule update --init --recursive`". |
| Runtime: `HiveStatus[name]` missing for an embedded site persona | Default to `"ready"`. Non-disruptive. Happens only for personas with no hive counterpart. |
| Runtime: DDL `ADD COLUMN` on existing DB | Idempotent via `IF NOT EXISTS`; same pattern as existing additive migrations. |
| CI: `go generate` output diverges from committed `status_gen.go` | `git diff --exit-code` fails the job. Author must run `go generate` locally and commit. |

## Testing

1. **Unit — generator**: `cmd/gen-persona-status/main_test.go` with a fixture dir containing three fake `.md` files (including one with no Status comment, which should be skipped). Assert generated Go file is byte-identical to a golden file.
2. **Integration — seeding**: extend or add a test in `graph/` that seeds personas into a test DB and asserts `Active=false` when `HiveStatus[name] ∈ {absorbed, retired}`.
3. **CI diff check** catches drift between committed `status_gen.go` and what the pinned hive commit would produce.

## Constitutional invariants

- **EXPLICIT** — Status is declared in exactly one place (hive `agents/*.md`). Site's copy is generated, never edited by hand.
- **BOUNDED** — `cmd/gen-persona-status` does one thing. `status_gen.go` has one purpose.
- **IDENTITY** — does *not* fix the "Status belongs on Role, not Persona" layer violation (the debt doc's long-term concern). It removes the *duplication* of Status, which is a prerequisite for the eventual Role/Persona split.
